# 采购选品分析（Procurement Analysis）

把采购分析当成**证据分级 + 参数归一 + 风险揭示**，不是把搜索结果堆成一张表。

核心原则：**先建立比较框架，再填数据；先找负面证据，再给推荐。**

## When to Use

Use when the user wants any of these outcomes:

- 产品横评、竞品分析、选品建议
- 供应商/品牌/型号对比
- 按预算推荐、按场景推荐
- “哪个好”“怎么选”“值不值得买”
- 需要参数表、推荐表、shortlist、采购结论

Do **not** use for:

- 纯聊天式主观偏好讨论，没有对比或采购意图
- 已经有完整、可信、单一官方数据源，只需要机械整理

## What This Skill Prevents

通用代理在这类任务里常见失误：

1. 把 **SEO 推荐文、论坛软文、品牌页、真实用户评价** 混成同权重证据
2. 看到一个价格或参数就直接入表，不标注时效和置信度
3. 只搜正向内容，不主动搜“缺点 / 翻车 / 维修 / 投诉”
4. 直接给“最佳推荐”，却没说明推荐是基于什么场景
5. 把 **和面机、厨师机、半功能产品** 混为一类，导致横评失真

这个 skill 的作用就是强制避免这些错误。

## Workflow

### 1. Define the comparison frame first

先明确 4 件事：

- **对象边界**：比较的是哪一类产品/供应商？
- **价格带**：主战场价位是多少？
- **关键维度**：哪些参数真正影响决策？
- **输出口径**：是消费建议、竞品分析，还是采购 shortlist？

如果用户没有明确说明，默认输出：

1. 核心参数说明
2. 横向对比表
3. 风险点/坑点
4. 按预算或场景推荐

### 2. Grade sources before trusting them

按这个顺序给信源分级：

| 等级 | 信源类型                           | 用法                           |
| ---- | ---------------------------------- | ------------------------------ |
| A    | 官方页面、官方规格书、官方售后政策 | 用于型号、规格、配件、保修     |
| B    | 主流电商商品页、历史价格页         | 用于价格、在售状态、用户关注点 |
| C    | 长篇横评、实测评测、论坛追评       | 用于体验、噪音、稳定性、缺点   |
| D    | 聚合推荐文、SEO 软文               | 只能辅助发现候选，不直接当结论 |

**规则：**

- 型号、参数、保修：优先 A/B
- 价格：优先 B，并标注“活动价/补贴价/常规价”
- 缺点和真实体验：优先 C
- D 级信源不能单独支撑结论

### 3. Normalize product types before comparing

先把候选分组，再横评：

- 真厨师机
- 和面机 / 单功能揉面机
- 伪同类（名字像，但用途不同）

如果不是同类，必须拆表，不能混评。

### 4. Force negative evidence collection

每个重点型号都必须主动搜一轮负面信息，至少覆盖：

- `型号 + 缺点`
- `型号 + 翻车`
- `型号 + 黑粉 / 噪音 / 晃动 / 维修`
- `型号 + 追评`

如果找不到负面信息，也要说明：**“当前未检索到充分负面反馈，不等于没有问题。”**

### 5. Mark uncertainty explicitly

对每个重要参数给出置信度：

| 标记     | 含义                       |
| -------- | -------------------------- |
| 已验证   | 至少两个高质量独立信源一致 |
| 单一信源 | 只有一个可信来源           |
| 口径冲突 | 不同来源存在矛盾           |
| 待确认   | 页面存在但信息不完整       |

不确定时不要“补一个看起来合理的数字”。

### 6. Recommend by scenario, not “one best product”

推荐必须按场景输出，例如：

- 最均衡
- 低预算首选
- 品牌成熟度优先
- 智能体验优先
- 大容量优先

**不要直接给单一“最佳”。**

## Output Format

默认按下面这个顺序输出：

### A. 结论先行

用 3-5 行说清楚：

- 当前主流候选是谁
- 各自适合什么人
- 当前最大的风险点是什么

### B. 核心参数表

```markdown
| 型号 | 价格 | 关键参数 | 主要优点 | 主要风险 | 置信度 | 主要信源 |
| ---- | ---: | -------- | -------- | -------- | ------ | -------- |
```

### C. 风险与坑点

至少覆盖：

- 参数口径冲突
- 活动价与日常价差异
- 产品类型混淆
- 常见负面反馈

### D. 推荐清单

```markdown
| 场景       | 推荐 |
| ---------- | ---- |
| 最均衡     | ...  |
| 低预算     | ...  |
| 品牌成熟度 | ...  |
| 特殊需求   | ...  |
```

### E. 预算边界说明

如果用户给了一个主预算带，补一句边界判断：

- **预算再降会牺牲什么**
- **预算再加能换来什么**

## Quick Reference

### 必做清单

- 先分产品类型，再横评
- 先列维度，再填参数
- 价格必须标时间性/活动属性
- 重要参数必须标置信度
- 表格里必须给出主要信源或来源类型
- 主动搜索缺点/翻车/维修
- 推荐必须按场景，不给单一“神机”
- 主预算带之外，补一句“再降/再加”的边界说明

### 推荐维度模板

适用于大多数消费品/设备采购：

- 价格与总持有成本
- 核心规格/性能
- 稳定性与真实使用反馈
- 售后与保修
- 配件/生态/扩展性
- 风险点与争议点

## Common Mistakes

| 错误                     | 为什么错                 | 正确做法                   |
| ------------------------ | ------------------------ | -------------------------- |
| 看到一篇推荐文就开始写表 | 候选会被营销投放带偏     | 先建候选池，再交叉验证     |
| 价格写得很精确但没日期   | 电商价格波动极大         | 标注“活动价/补贴价/常规价” |
| 不标信源等级             | 读者会误以为结论同样可靠 | 区分 A/B/C/D 级信源        |
| 不搜负面反馈             | 输出会天然偏正向         | 每个重点型号都搜缺点/翻车  |
| 混评不同类型产品         | 横评结论失真             | 先分组，再横评             |
| 直接说“最佳推荐”         | 把不确定性藏起来了       | 按预算/场景给推荐          |

## Minimal Example

用户说：

> 帮我调研一下 1000 元左右的厨师机，列举核心参数，做成竞品分析表格。

正确做法不是直接列 5 个型号，而是：

1. 先限定比较对象：真厨师机 vs 和面机是否分开
2. 先定关键维度：电机类型、容量、功能完整度、稳定性、价格
3. 再去搜：官方参数、在售价格、真实缺点
4. 表格中标记：哪些参数已验证，哪些口径冲突
5. 最后按场景推荐：最均衡、低预算、智能体验、品牌成熟度

## Final Rule

**采购分析不是“把搜到的内容排整齐”，而是“把证据质量和决策逻辑讲清楚”。**

如果你不能说明：

- 为什么这些候选被放进同一张表
- 为什么这些维度值得比较
- 为什么这个推荐适用于这个场景

那就还没完成真正的采购分析。
