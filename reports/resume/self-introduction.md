# 面试自我介绍

面试官您好，我是张三，7 年运维开发经验，现在主要做 Go 基础设施和 AIOps 方向。

我的工作可以概括成一句话：**让业务开发更高效，让 AI 能力更易用。**

在公司，我负责的统一认证网关接入了 100 多个服务，是全公司的服务入口。同时我也沉淀了一整套基础库——MQ、缓存、日志、配置——团队十几个人都在用。我不只是写代码，更注重工程化：标准化流程、开发规范、最佳实践文档，包括设计背后的 WHY，而不只是 HOW。

最近一年我在做 AIOps 方向的探索，写了几个 AI Agent 技能：自动巡检、代码审查、方案评审。还做了一个 GitLab AI 机器人，重点解决了跨项目安全隔离的问题——通过 Pipeline + 自建 Runner + Project Access Token，让 Agent 只有项目级权限，杜绝越权风险。

**说一个我的特点：我习惯先定义问题，再动手。**

比如做框架这件事，团队其实大部分人还是在用开源框架，我也没强制替换。但我会想清楚：我们团队到底需要解决什么问题？是新人上手慢？是 Code Review 没标准？还是技术栈不统一？找到问题本质，再决定是写框架、写规范、还是写文档。不是为了做而做。

有人可能会觉得，内部项目没什么技术含量，规模也不大。但换个角度：
- 内部项目意味着要理解多个业务线的需求，设计通用方案，这比单一业务开发更考验架构能力
- 规模不大意味着每一行代码、每一个设计决策都不能浪费，反而需要更精准
- 而且基础设施出问题影响的是全公司，这种压力下锻炼出来的稳定性意识，是写业务代码很难获得的

我平时也喜欢折腾，家里有服务器和 GPU，自己跑模型、搭 GitLab、搞网关，运维相关的我都自己动手。不是为了卷，就是觉得**只有自己用起来，才知道工具哪里不好用**。

以上就是我的介绍，谢谢。

## 「劣势变优点」对照

| 可能被质疑的点 | 怎么转 |
|--------------|--------|
| 都是内部项目，没业务指标 | 内部项目考验通用方案设计能力，影响范围是全公司 |
| 规模不大，几千人 | 规模小 = 每个决策都不能浪费 = 更精准的架构能力 |
| 框架没人强制用 | 不强制 = 务实，解决问题靠规范和文档，不靠行政命令 |
| 不怎么用 LangChain 这些框架 | 框架不是重点，定义问题才是；自己写更可控 |
| 基础设施听起来不像核心业务 | 基础设施是水电煤，出问题全公司停摆 |
